## 1. 项目概况

基于 **Three.js (r0.169) + Vite** 的浏览器端轻量品牌驾驶游戏（免安装，单文件可分发）。原型由开发者 "Fable" 搭建，现持续迭代中。

- 核心玩法：海岛自由漫游 + 三条竞速路线（金银铜奖牌 + 成就）、可破坏道具、漂移/腾空物理、动态昼夜天气、照片模式+海报导出、Lofi 电台。
- 双模式：**驾驶**（第三人称追车）与 **步行**（可控角色，过肩视角，F 键切换）。
- 角色：**Cici**（轻盈）与 **IRON**（硬核，多动作），车库可选人。
- 在线体验：https://horizon.beastle.cn

### ⚠️ 关键工作约束
开发者**无法运行本地 dev server**，所有画面 review 都通过**单文件 HTML 预览**（`地平线海岸_预览.html`，约 39MB，所有资源 base64 内联）。任何改动都必须重建该预览文件给开发者确认。详见 §5。

## 4. 架构与模块职责

启动流程（`main.js` 末尾的异步 IIFE）：分阶段调用 `buildTerrain → buildRoad → buildScenery → buildEnv → buildProps → buildCharacter → buildSkyCycle`，再 `initUI / initFX / settleCarPose / applyTod / enterGarage / loop()`。每阶段间用 `requestAnimationFrame` 让出主线程刷新启动进度条，消除黑屏。

主循环 `loopBody()`（`main.js`）按 `G.appState` 分支：
- `drive`：`physics → raceUpdate → gameplayUpdate → fxUpdate → updateChaseCamera`
- `walk`：`characterUpdate → characterCamera → coastVehicle`（无人车自然滚停）
- `garage`：`controls.update`(轨道) + `characterPreviewUpdate`(车库站立 idle)
- 每帧都跑：`skyCycleUpdate`、`audioUpdate`、阴影/补光跟随焦点、`updateCarReflection`(反射探针)、`composer.render()`
- 健壮性：整个 loopBody 包 try/catch（异常只丢帧不冻死）；看门狗检测连续超长帧自动急停。

全局可变状态集中在 `core.js` 的 `G`（appState/camMode/hiQuality/weatherOn/skinIdx/curTod/charId/headlights/lampMats…）。

模块依赖（简）：`main` → 全部；`ui ↔ character`、`ui → vehicle/world/gameplay`；`vehicle/skycycle → world/core`；`world → core`。注意 `character.js` 现在 import 了 `vehicle.js` 的 `state`（用于车库预览定位）。

关键数据流：
- **道路中线**（`world.js` 的 `samples/tangents/normals`）是地形塑形、车辆贴地、竞速、小地图的统一数据源。
- **`curSunDir`**（world 导出）由 `skycycle` 每帧写入，`main` 据此摆放方向光/补光。

## 5. 构建与预览（★ 最重要）

### 5.1 标准开发（需要本机能跑 dev server）
```bash
npm install
npm run dev      # localhost 开发
npm run build    # → dist/
```

### 5.2 单文件预览（开发者唯一的 review 方式）
开发者看不到 dev server，只能打开一个**自包含的 HTML**。重建命令：
```bash
bash tools/build-preview.sh
```
该脚本做三件事：
1. `node tools/gen-assetmap.mjs` —— 把 `public/assets` 下所有**运行时按路径加载**的资源（glb/hdr/贴图/trees）转成 base64 data URI，写到 `/tmp/assetmap.json`。
2. `npx vite build --config vite.singlefile.mjs --outDir /tmp/hc-single` —— `vite.singlefile.mjs` 里的自定义 `inlineAssets` 插件在 transform 阶段**用字符串替换**把源码里的资源路径字面量替换为 data URI（例如 `'assets/character.glb'`、`'assets/iron.glb'`、`'assets/sky/' + k.f + '.hdr'`、`TP+'sand_diff.jpg'` 等），再由 `viteSingleFile` 把 JS/CSS 全部内联进一个 HTML。
3. 拷贝产物到 `地平线海岸_预览.html`。

⚠️ **内联替换是字面量精确匹配**：如果你改了源码里这些资源路径的写法（比如把 `'assets/iron.glb'` 拆成变量拼接），对应的 `vite.singlefile.mjs` 替换规则也必须同步改，否则该资源不会被内联，单文件里会加载失败。新增需要内联的资源时：①加进 `tools/gen-assetmap.mjs`；②在 `vite.singlefile.mjs` 加替换规则。

✅ 常规重建**不需要联网、不需要安装**（vite 与插件已在 node_modules，随项目持久化；assetmap 每次从本地资源现生成）。`/tmp` 被清也没关系，脚本会重生成。

## 6. 资产管线

### 6.1 角色（Mixamo FBX → GLB）—— 见 `Mixamo转GLB与角色集成规范.md`
**必须用 `fbx2gltf`（FBX2glTF 二进制），不要用 assimp**：assimp 会引入 `$AssimpFbx$` pivot 节点破坏 Mixamo 蒙皮，导致角色炸裂/倒置。流程：
1. 每个动作 FBX 单独 `FBX2glTF -i x.fbx -o x --binary`。
2. 用 `@gltf-transform/core` 以 idle 为基底，按**节点名**把其它动作的动画通道合并进同一文档（重定向到基底骨架）。
3. **剥离 Hips 的 X/Z 位移**（保留 Y），让动作原地播放（位移交给游戏逻辑）。
4. `dedup + prune + textureCompress(webp)` 优化，输出单个多动画 GLB。

这些离线工具（fbx2gltf/gltf-transform/sharp）**未装进项目**，需要时临时 `npm i` 到临时目录使用。

### 6.2 地形 PBR
Poly Haven 风格 PBR：每种地表 `diff(jpg) + nrm/rough(webp)`。`world.js` 用 `MeshStandardMaterial.onBeforeCompile` 做 4 向 splat（sand/forest/rock/dry 按高度/坡度权重在顶点着色器算，共享一张法线）。道路用 Road009B（`road2_*`）。

### 6.3 树木
`bake-trees.mjs`（EZ-Tree）离线烘焙 → `trees.json/bin` + 各树种色图。运行时实例化。

## 8. ★ 已知问题与历史决策（接手前必读）

### 8.1 动态天空贴图循环 —— **进行中，未确认修复**
- 现象：开发者反复反映"白天黑夜永远只有一张天空贴图，不循环"。
- 调试已确认：循环逻辑在跑（phase 推进、`scene.background` 已是动态源、5 张 HDR 都正确加载并能被 RGBELoader 解析），但**背景图始终停在白天**。
- 历史：天空经历过 dome 双球 → blendRT 离屏混合 → 程序化大气(THREE.Sky，被否：画质差+性能崩，已回退) → 回到 HDR。
- **当前实现**（`skycycle.js`）：已**弃用 blendRT 离屏混合**，改为**直接 `scene.background = texs[i]`**（当前时段 HDR，硬切换，无平滑过渡）。此改动**尚未被开发者确认是否生效**。
- ⚠️ `skycycle.js` 里**仍有屏幕调试条 `_dbg`**（顶部绿字）和 try/catch 包裹 + console 日志——**确认天空修好后必须移除这些调试代码**。
- 待办：①确认硬切换能正常昼夜变化；②若可行，用更稳的方式把**平滑交叉淡入**加回来（建议：两张背景球 mesh 叠加用 opacity 交叉淡入，避免 blendRT 那套离屏 RT 的坑）。
- 5 张 HDR 实测亮度（供调参）：day2 偏阴天、day3 有强太阳(峰值 5万)、evening 峰值~9、night1/night2 很暗。注意 day3 的太阳尖峰会在镜面/PMREM 上溢出（"方块太阳"），历史上用 `clampMat` 把反射环境亮度钳到 12 解决。

### 8.2 车辆光照/反射（已回退到"昨天的好版本"，勿轻动）
- 本会话曾反复改车辆反射钳制/玻璃材质，导致车身死黑、泛白、塑料感等连环 regression，**已全部回退**到：反射探针直接反射真实场景（未钳制）、玻璃为原厂深色介质镜面（`envMapIntensity 1.8, roughness 0.06, metalness 0`）。
- 遗留（可接受/未修）：①步行/俯视时太阳在车漆/玻璃上的高光偏强（最初诉求"反射太晃眼"未根治）；②正俯视车顶玻璃因菲涅尔(正面仅~4%反射)+深色底色显得偏黑——这是介质玻璃的物理特性。
- **教训：光照/反射高度耦合（反射探针 + 环境钳制 + Bloom + 曝光 + 材质）。一次只改一个参数并立即重建确认，不要批量改。**

### 8.3 IRON 角色材质
- IRON 的 Mixamo 导出**只引用了一张法线贴图（且源文件缺失），没有任何颜色/粗糙度贴图**。转换时坏法线图被塞成 96 字节空图，曾导致"破面"。已**移除坏法线图**，目前材质为干净的**钢灰单色**（衣服 metalness 0.08/rough 0.78，面罩光泽黑）。
- 若要 IRON 高质感，需补一套真实 PBR 贴图（base color / roughness / metallic / normal）。

## 9. 渲染/性能注意

- 色调映射 ACES，基础曝光 1.35（运行时由 skycycle 按时段覆盖）。
- **GTAO 默认关闭**（`core.js`，`G.aoOn` 控制）：会与透明烟雾/粒子冲突产生黑块、并在地面产生随视角移动的白膜，且开销近翻倍。改用**烘焙 AO**（地形顶点色）。
- Bloom 半分辨率计算；阴影 2048、视锥 ±90、far 900。
- 动态像素比：停车 1.5 / 高速 1.2；FPS<42 时一次性 `setQuality(false)` 自动降画质。
- 海面用低粗糙 PBR 平面 + 法线波纹 + 环境反射（弃用 Three.js Water 实时平面反射，省 ~80% 该项开销）。

## 11. 踩坑清单（节省接手时间）

1. **沙箱 `/tmp` 重启即清空**：构建脚本已做成从本地资源现生成 assetmap，常规重建无需重装。角色转换工具（fbx2gltf 等）每次需临时重装。
2. **单文件内联是字面量匹配**：改资源路径写法要同步改 `vite.singlefile.mjs`。
3. **改完必重建预览**给开发者看（`bash tools/build-preview.sh`），开发者无法跑 dev。
4. **角色用 FBX2glTF 不用 assimp**。
5. **动画"瞬移/抽搐"** 多为没剥 Hips 根位移。
6. **WASD 方向**：相机相对移动，右向量为 `(-cos, sin)`（曾因符号错导致"按左走右"）。
7. **光照别批量改**：耦合极深，逐项改+重建确认。
8. 预览文件约 39MB（HDR 内联）；若改回程序化天空可降到 ~19MB，但实测画质/性能更差，已否决。
9. 回退参考"好版本"时，本会话的安全基线 = 车辆反射/玻璃回到原厂设定、天空回到 HDR。
